# 進度與需求.md 欄位定義

本文檔詳細說明 `進度與需求.md` 中各欄位的定義、填寫方式與範例。

---

## 1. 專案基本資訊

### 專案名稱
- **定義**：專案的簡潔名稱
- **範例**：`電商會員管理系統`、`內容管理平台升級`

### 專案目標
- **定義**：專案要達成的核心目標（1-3 句話）
- **範例**：
  - 建立一個支援 10 萬並線使用者的會員系統
  - 整合現有的使用者資料庫，提供統一的認證機制

### 專案範圍
- **定義**：專案包含什麼、不包含什麼
- **範例**：
  - **包含**：使用者註冊、登入、權限管理、密碼重置
  - **不包含**：社群登入（Google/Facebook）預計 v2.0 加入

### 專案狀態
- **定義**：專案當前的整體狀態
- **選項**：🔴 待開始 / 🟡 進行中 / 🟢 已完成 / ⚪ 已暫停

## 2. 需求清單

### 功能需求 ID 命名規則
- 格式：`FR-NNN`（FR = Functional Requirement）
- 範例：`FR-001`, `FR-002`

### 優先級定義

| 等級 | 名稱 | 說明 | 處理原則 |
|------|------|------|----------|
| P0 | 必須有 | 核心功能，沒有它專案失去意義 | 不可延後，必須完成 |
| P1 | 重要 | 重要功能，影響主要使用場景 | 儘早完成 |
| P2 | 次要 | 有用但非必要的功能 | 可排到後期 |
| P3 | 選配 | 錦上添花的功能 | 有資源再做 |

### 非功能需求類別
- **效能**：回應時間、處理能力、並發數
- **安全**：加密、認證、授權
- **可用性**：系統可用性比例、故障恢復時間
- **相容性**：瀏覽器、裝置、作業系統支援

### 狀態標籤

| 標籤 | 意義 | 使用時機 |
|------|------|----------|
| 🔴 待開始 | 尚未開始處理 | 需求已確認，等待開發 |
| 🟡 進行中 | 正在處理 | 已有人負責，正在實作 |
| 🟢 已完成 | 完成並驗收 | 功能完成並通過驗收 |
| ⚪ 已取消 | 不再需要 | 需求變更或不再需要 |
| 🔵 需討論 | 需要進一步討論 | 規格不清，需要確認 |

## 3. 使用者故事

### 標準格式
```
作為 [角色]，我想要 [功能]，以便 [價值]
```

### 範例
- 作為**系統管理員**，我想要**查看所有使用者的登入記錄**，以便**追蹤異常活動**
- 作為**一般使用者**，我想要**重置我的密碼**，以便**在忘記密碼時能重新存取帳戶**

### 驗收標準 (Acceptance Criteria)
- 定義「完成」的具體條件
- 範例：
  1. 使用者可透過 Email 收含重置連結
  2. 重置連結有效期為 30 分鐘
  3. 重置後舊密碼立即失效

## 4. 規格書

### 技術架構
- 說明專案使用的主要技術棧
- 包含：Frontend、Backend、Database、Infrastructure

### 系統架構
- 描述系統的分層結構
- 前端層 → 後端層 → 資料層 → 外部服務

### API 介面範例

| 欄位 | 說明 | 範例 |
|------|------|------|
| 端點 | API 路徑 | /api/users |
| 方法 | HTTP 方法 | GET / POST / PUT / DELETE |
| 描述 | API 功能說明 | 取得使用者清單 |
| 參數 | 請求參數 | page, limit, search |
| 回傳 | 回傳資料型別 | User[] |
| 狀態 | 開發狀態 | 🔴 待開始 / 🟢 已完成 |

### 資料結構
- 使用 TypeScript Interface 定義資料模型
- 包含欄位名稱、型別、說明

### 資料庫設計

| 欄位 | 說明 | 範例 |
|------|------|------|
| Table | 資料表名稱 | users |
| 欄位 | 欄位名稱 | id, name, email |
| 型別 | 資料型別 | UUID, VARCHAR(100) |
| 說明 | 欄位說明 | 主鍵、姓名 |
| 索引 | 索引類型 | PK (主鍵), UK (唯一), IDX (一般) |

## 5. 測試計畫

### 測試類型

| 類型 | 說明 | 常用工具 |
|------|------|----------|
| 單元測試 | 測試個別函數/元件 | Jest, Vitest |
| 整合測試 | 測試 API 介面 | Supertest, Postman |
| E2E 測試 | 測試完整使用者流程 | Playwright, Cypress |
| 效能測試 | 測試系統效能 | k6, JMeter |

### 測試案例 ID 命名規則
- 格式：`TC-NNN`（TC = Test Case）
- 範例：`TC-001`, `TC-002`

### 測試報告欄位

| 欄位 | 說明 |
|------|------|
| 測試日期 | 執行測試的日期 |
| 測試類型 | 單元測試、整合測試、E2E 測試 |
| 測試數量 | 總測試案例數 |
| 通過/失敗 | 測試結果統計 |
| 覆蓋率 | 程式碼覆蓋率百分比 |
| 報告連結 | 完整報告的連結 |

## 13. 工作日誌

### 必填欄位

| 欄位 | 說明 | 範例 |
|------|------|------|
| 日期時間 | 格式：YYYY-MM-DD HH:MM | 2026-02-01 14:30 |
| 任務 | 執行的工作項目 | 實作使用者登入功能 |
| 完成內容 | 具體完成的細項 | • 完成 JWT 驗證邏輯<br>• 新增 API 端點 |
| 修改檔案 | 變更的檔案列表 | `src/auth/login.js` |
| 狀態 | 當前狀態 | ✅ 完成 |
| 備註 | 其他需要記錄的事項 | 遇到的問題、學到的經驗 |

### 狀態標籤（工作日誌）

| 標籤 | 使用時機 |
|------|----------|
| ✅ 完成 | 順利完成，無待辦事項 |
| 🟡 進行中 | 部分完成，還需要繼續 |
| ❌ 有問題 | 遇到阻礙，需要處理 |
| ⏸️ 暫停 | 暫時停止，等待某事項 |

## 14. 版本記錄

### 版本號規則
- `v0.1`：初始版本
- `v0.2`, `v0.3`：需求新增或修改
- `v1.0`：第一版正式完成

### 變更原因範例
- **新增功能**：用戶要求增加社群登入
- **需求變更**：原本 P2 需求提升為 P1
- **規格調整**：API 介面參數調整
- **問題修復**：修正前一版本的錯誤
